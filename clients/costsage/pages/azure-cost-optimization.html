Azure Cost Optimization: A 2026 FinOps Playbook

TL;DR

Azure has six high-leverage cost levers: Reservations, Savings Plans for compute, Azure Hybrid Benefit, right-sizing, idle cleanup, and Spot. Most engineering teams use one or two and leave 30-50% of possible savings on the table. The fix is not another report — it is closed-loop execution that ties detection to action under operator approval.

Why Azure cost is harder than it looks

Azure has the deepest discount surface of any major cloud provider — and arguably the most fragmented. Customers can stack Reservations, Savings Plans, Hybrid Benefit, dev/test pricing, Enterprise Agreements, and Azure-specific commitments into a single bill that few finance teams can fully decompose. The complexity is the entire reason Azure cost overshoots happen so often: the discount the team thought they were getting is not the discount actually applied.

This playbook walks through the six levers in order of typical savings impact and shows how to operate each one in 2026.

Lever 1: Azure Reservations

Azure Reservations are 1-year or 3-year commitments to a specific instance size, region, and (optionally) operating system. Discounts range from 40% to 72% off pay-as-you-go. Reservations cover VMs, SQL Database, Cosmos DB, Synapse, App Service, and a handful of other services.

The right way to operate them: cover the steady, predictable base of your workload — never the autoscaling tail. Most teams err on the side of caution and end up under-bought; a smaller minority over-buy and end up paying for unused commitments. The middle path is to reserve only what has been steady for 6+ months at the instance size you're confident will not change.

CostSage models reservation coverage by workload and runs a what-if planner — what does my bill look like at 60% coverage vs 80% — before any commitment is made.

Lever 2: Savings Plans for compute

Azure Savings Plans (introduced in 2022) are a $/hour commitment that applies across VM families, regions, and OS types. Discounts are smaller than Reservations (up to 65%) but the flexibility is much higher — useful for engineering teams that frequently change VM sizes.

Decision rule: if your VM mix is stable, use Reservations. If your VM mix changes more than quarterly, use Savings Plans. Many mature Azure customers run a hybrid: Reservations on the most stable workloads (e.g. SQL servers, AD controllers) and Savings Plans on everything else.

Lever 3: Azure Hybrid Benefit

Azure Hybrid Benefit (AHB) is the largest single discount available on Azure for migrated workloads. Customers with on-premises Windows Server and SQL Server licences (with active Software Assurance) can apply those licences to Azure VMs and SQL resources — Azure removes the licence component from the rate. Combined with a 3-year Reservation, AHB can drive total savings on a Windows VM to over 80%.

The operational gap: many customers own AHB-eligible licences and forget to apply them. The Azure portal does not aggressively surface this. Audit every Windows VM and SQL resource quarterly for AHB applicability — CostSage runs this check on every refresh.

Lever 4: Right-sizing

Right-sizing is the discipline of matching VM size to actual usage. Azure Advisor surfaces basic recommendations, but it has a conservative threshold — many opportunities are not flagged. A more aggressive operating rule: any VM under 40% peak CPU and under 30% peak memory over 14 days is a candidate to drop one tier or move to a Bs-series burstable family.

Right-sizing is the single highest-leverage lever for newer Azure tenants where reservation coverage hasn't been built yet. It also stacks: a right-sized VM under a Savings Plan or Reservation compounds savings.

Lever 5: Idle resource cleanup

Common idle waste on Azure:

  • Unattached managed disks (created during VM resize, never reattached)
  • Static public IPs reserved but not assigned
  • VMs in stopped (not deallocated) state — still billing for compute
  • Empty App Service plans
  • Orphaned NSGs, NICs, and snapshots
  • Dev/test VMs left running over weekends
  • Old ASE (App Service Environment) instances

None of this is sophisticated, but the cumulative cost is large. A medium-sized Azure tenant typically carries 5-12% of monthly spend in pure idle waste. Run an idle audit monthly. CostSage runs it continuously and queues a deletion plan for operator approval.

Lever 6: Spot VMs

Azure Spot Virtual Machines offer up to 90% off pay-as-you-go in exchange for the right to evict with 30 seconds' notice. Suitable workloads: batch processing, build/CI agents, dev/test environments, big-data jobs with checkpointing, and stateless workloads behind a queue.

Not suitable: stateful databases, customer-facing web tiers, anything with strict uptime SLOs. Pair Spot with VMSS (scale sets) and a max-eviction-rate policy to balance savings against availability.

Allocation: the precondition for everything

Every lever above depends on knowing what each Azure resource costs and which team or product owns it. The Azure tagging story is workable but requires discipline: enforce a tag policy (Azure Policy) that requires CostCenter, Product, Environment, and Owner on every resource. Without this, all the cost recommendations in the world end up as orphaned tickets nobody owns.

CostSage augments tag-based allocation with workload-aware grouping — useful when historical resources are tagless and rebuilding tags is impractical.

Common Azure cost mistakes

  • Running VMs in the wrong region. Azure prices vary materially between regions for the same VM size. East US 2 is generally cheaper than UK South for equivalent compute. If latency permits, region choice can shave 10-20% off baseline cost.
  • Premium SSD by default. Standard SSD is sufficient for most workloads. Premium SSD costs 2-3x and is reflexively chosen when "production." Audit disk SKUs against actual IOPS need.
  • Forgotten dev/test subscriptions. The dev environment that never got cleaned up after a project was cancelled. Tag-policy + monthly idle review catches this.
  • Egress surprises. Cross-region replication, AKS pulling images from another region, ExpressRoute mis-configuration. Network egress on Azure is one of the least-tracked cost categories.
  • App Service over-provisioning. Premium v3 plans assigned to low-traffic web apps. Often a single P1v3 covers what was previously distributed across multiple S1 plans — without a downgrade audit, the savings never materialise.
  • SQL DTU vs vCore mis-match. The older DTU model and the newer vCore model price differently for the same workload. Re-evaluate annually.

Reservations vs Savings Plans: a worked example

Consider a team running a mix of D-series and E-series VMs that totals ~$12,000/month on Azure compute. Suppose 70% of that is steady production and 30% is dev/test that fluctuates with project mix.

  • Pure on-demand: $12,000/month. No discount.
  • 3-year RIs covering the steady 70%: ~$8,400 baseline at ~55% discount → $3,780, plus the dev/test 30% on demand at $3,600 → ~$7,380/month. Caveat: requires VM mix to remain stable; if D-series flips to E-series mid-term, RI flexibility helps but cross-family conversion has limits.
  • 3-year Savings Plan covering ~70% of compute spend: ~$8,400 at ~55% discount → ~$3,780, plus the variable 30% on demand at $3,600 → ~$7,380/month. Same total, but with significantly more flexibility on VM-size/family changes.

The choice between RI and SP is not about absolute savings (often similar) — it's about how much the VM mix is expected to shift. Most teams over-estimate stability and under-buy SPs.

FAQ

What is Azure cost optimization?

Azure cost optimization is the discipline of reducing Microsoft Azure spend without harming reliability or performance. Core levers include Azure Reservations, Savings Plans for compute, Azure Hybrid Benefit, right-sizing virtual machines, decommissioning idle resources, and applying Spot VMs to interruption-tolerant workloads.

What is the difference between Azure Reservations and Savings Plans?

Reservations commit to a specific VM size or service for 1 or 3 years and offer the deepest discounts (up to 72%) but require workload stability. Savings Plans for compute commit to a $/hour spend across compute services and provide flexibility (up to 65%) — useful when VM sizes change frequently.

What is Azure Hybrid Benefit?

Azure Hybrid Benefit lets customers apply existing on-premises Windows Server and SQL Server licences (with Software Assurance) to Azure VMs and SQL services, reducing the Azure rate by removing the licence component. It is one of the largest single discounts available on Azure for migrated workloads.

How do I find idle resources on Azure?

Use Azure Advisor's cost recommendations, Cost Management's anomaly detection, and Resource Graph queries that surface VMs at low CPU and network for over 14 days, unattached managed disks, idle public IPs, and orphaned snapshots. CostSage automates this across subscriptions and queues actions for operator approval.

Are Spot VMs reliable for production?

Spot VMs offer 60-90% discounts but can be evicted with 30 seconds' notice. They are appropriate for batch processing, build agents, dev/test, and stateless workloads with checkpointing. They are not appropriate for stateful production services.
